# 公共能力接口约定 V1

状态：首版冻结  
适用成员：A/B/C/D  
原则：业务模块只依赖本文件中的稳定路径、字段和事件名；内部实现可由成员 A 后续替换。

## 1. 通用规则

- API 基础路径：`/api`。
- 登录请求携带 `Authorization: Bearer <token>`。
- 成功响应：`{"code":0,"message":"success","data":...,"timestamp":"..."}`。
- 常用错误码：`40001` 参数错误、`40100` 未登录、`40300` 无权限、`40400` 数据不存在、`40900` 状态冲突、`50000` 系统错误。
- 业务表只保存附件 `id`，不保存本机路径或临时 URL。
- 字典和权限使用英文编码作为稳定标识，中文名称只用于展示。

## 2. 附件接口

### 上传

`POST /api/attachments`，请求类型 `multipart/form-data`：

- `file`：文件内容，必填。
- `businessType`：业务类型，如 `GIFT_APPLY`、`SUBSIDY_APPLY`、`WORK_STUDY`。
- `businessId`：业务主键；草稿阶段可空，提交后通过绑定接口补充。

响应 `data`：

```json
{
  "id": 1001,
  "originalName": "证明.pdf",
  "contentType": "application/pdf",
  "size": 102400,
  "status": "READY"
}
```

### 查询、下载与删除

- `GET /api/attachments/{id}`：获取元数据。
- `GET /api/attachments/{id}/content`：鉴权后下载或预览。
- `DELETE /api/attachments/{id}`：仅草稿或未绑定附件允许删除。
- `PUT /api/attachments/{id}/binding`：绑定业务，正文 `{"businessType":"...","businessId":123}`。

限制：首版单文件不超过 20 MB；允许 PDF、PNG、JPG、JPEG、XLSX、DOCX；服务端必须校验真实 MIME 类型和访问权限。

## 3. 字典接口

- `GET /api/dictionaries/{dictTypeCode}`：获取启用字典项，所有登录用户可读。
- `GET /api/system/dictionaries`：分页管理字典，权限 `system:dictionary:view`。
- `POST /api/system/dictionaries/items`：新增字典项，权限 `system:dictionary:edit`。
- `PUT /api/system/dictionaries/items/{id}`：修改字典项。
- `DELETE /api/system/dictionaries/items/{id}`：逻辑删除字典项。

字典项响应：

```json
{
  "itemCode": "2",
  "itemName": "待辅导员审核",
  "sort": 20,
  "enabled": true
}
```

缓存规则：后端缓存 5 分钟；管理端变更后立即失效。业务库只存 `itemCode`，不得存中文名称。

## 4. 操作日志接口

业务成员无需手写日志表插入语句，统一使用注解：

```java
@OperationLog(module = "困难补助", action = "SUBMIT", targetId = "#request.id")
```

日志自动记录操作者、模块、动作、目标主键、请求路径、结果、IP 和时间；密码、令牌、身份证号、银行卡号、文件内容必须脱敏或排除。

- `GET /api/system/operation-logs`：分页查询，权限 `system:log:view`。
- 查询参数：`module`、`operatorId`、`action`、`startTime`、`endTime`、`success`。
- 日志只允许查询和归档，不提供普通删除接口。

## 5. 消息接口与事件

业务模块通过事件发布消息，不直接写 `gc_message`：

```java
messagePublisher.publish(new MessageEvent(
    "SUBSIDY_REVIEW_PASSED", receiverUserId, businessId, variables));
```

首批冻结事件名：

- `GIFT_APPLY_SUBMITTED`
- `GIFT_REVIEW_RETURNED`
- `GIFT_REVIEW_PASSED`
- `SUBSIDY_APPLY_SUBMITTED`
- `SUBSIDY_REVIEW_RETURNED`
- `SUBSIDY_REVIEW_PASSED`
- `PAYMENT_STATUS_CHANGED`
- `WORK_STUDY_HIRED`
- `WORK_STUDY_AGREEMENT_PENDING`

用户接口：

- `GET /api/messages`：分页查询，参数 `readStatus`、`messageType`。
- `GET /api/messages/unread-count`：未读数。
- `PUT /api/messages/{id}/read`：标记单条已读。
- `PUT /api/messages/read-all`：全部已读。

消息事件必须在业务事务提交成功后处理，失败进入重试记录，不能因为通知失败回滚已经完成的审批。

实现约束：

- `MessagePublisher.publish` 是其他业务模块唯一入口；即使模板、数据库或渠道发送失败，也不向业务调用方抛出通知异常。
- 站内信持久化后分别调用 `SMS`、`WECHAT`、`EMAIL` 适配器；当前阶段均为模拟适配器，只记录渠道和消息主键，不记录接收人或正文。
- 渠道失败写入 `gc_message_delivery` 并标记 `RETRY`；事件生成失败写入 `gc_message_event_retry`，两类记录均只保存失败类型，不保存异常原文。

## 6. 变更规则

- 路径、字段、权限码和事件名的破坏性修改必须由 A 发起评审，B/C/D 至少两人确认。
- 新增可选字段允许直接扩展；删除字段、改类型、改语义必须提升接口版本。
- 调用方先基于本约定使用 mock；实际接口完成后只替换数据源，不重写页面。
