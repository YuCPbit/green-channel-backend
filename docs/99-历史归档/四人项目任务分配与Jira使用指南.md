## 1. 分工原则

1. 成员 A、B、C、D 先作为占位符，在 Jira 中替换为真实姓名。
2. 每人负责自己模块的数据表、后端接口、前端页面、自测和接口说明，尽量减少跨人等待。
3. 公共框架由 A 主责，但 B/C/D 必须参与评审，不得把公共问题全部丢给 A。
4. 每个模块必须形成“可运行的完整链路”，不能只有页面或只有接口。
5. 所有工作进入 Jira，不用口头约定代替任务卡。

## 4. Jira 应该怎么分类

### 4.1 推荐层级

```text
项目 Project
├─ 史诗 Epic：一大块业务目标
│  ├─ 故事 Story：从用户角度表达的可验收功能
│  │  ├─ 子任务 Sub-task：前端/后端/数据库/测试等实施步骤
│  └─ 任务 Task：不直接对应用户功能的技术或文档工作
└─ 缺陷 Bug：已实现功能中发现的问题
```

简单理解：

- Epic 是“一组功能”，不是一天内能做完的小事。
- Story 是“用户能做什么”。
- Task 是“团队为实现项目要做什么”。
- Sub-task 是 Story/Task 下的具体执行项。
- Bug 是对已有行为与预期不一致的记录。

### 4.2 建议新建的 8 个史诗

| Epic 名称 | 包含模块 | 主责人 |
| :--- | :--- | :--- |
| EP01 基础平台与新生信息 | 3.1、3.17、3.18 | A |
| EP02 绿色通道与大礼包 | 3.2—3.8 | B |
| EP03 困难补助双入口申请 | 3.9、3.10 | C |
| EP04 资助方案、资金与申诉 | 3.12、3.13、3.15 | B 负责方案/申诉，C 负责资金 |
| EP05 消息通知中心 | 3.11 | A |
| EP06 辅导员事务申请 | 3.19 | A 负责类型配置，C 负责申请/审批 |
| EP07 勤工助学 | 3.20 | D 主责，A/B/C 分担独立 Story |
| EP08 数据看板与报表 | 3.14、3.16 | D |

老师要求“新建史诗”时，就在 Jira 里把上表每一行建成一个 Epic。Epic 不要只填一个名字，还应填目标、范围、不包含什么和验收条件。

## 5. Jira 建立步骤

Jira 不同版本的菜单位置可能略有差异，但需要创建的内容相同。

### 第一步：创建项目

建议配置：

- 项目名称：`高校绿色通道系统`
- 项目键：`GCS`
- 模板：Scrum（如果老师仅要求看板，Kanban 也可以；本计划推荐 Scrum）
- 项目类型：如可选，四人小组选“团队管理”更简单；如老师要求统一工作流，选“公司管理”。
- 项目负责人：组长或成员 A。

### 第二步：邀请成员

- 将四人全部加入项目。
- 确认每个人都能创建、编辑、分配和移动任务。
- 不要共用同一 Jira 账号，否则无法看出个人工作量。

### 第三步：创建版本/发布

如 Jira 提供 Releases/Versions，建立：

- 版本名：`V1.0-课程验收版`
- 开始日期：2026-07-15
- 发布日期：2026-07-24
- 说明：包含绿色通道、礼包、补助双入口、辅导员事务、勤工助学、资金、看板和系统管理。

所有本次要做的 Story/Task/Bug 都把 Fix Version 填为这个版本。

### 第四步：创建 Epic

按第 4.2 节创建 8 个 Epic。每个 Epic 建议填：

- Summary：例如`EP03 困难补助双入口申请`。
- Description：说明学生自主申请和辅导员主动申请的业务目标。
- Assignee/Owner：主责人。
- Start date：7 月 15 日。
- Due date：7 月 23 日。
- Fix Version：`V1.0-课程验收版`。
- Labels：例如 `subsidy`、`core`。

### 第五步：在 Epic 下创建 Story

Story 不要写成“写补助代码”，应该用用户视角。例如：

`作为辅导员，我希望为所管学生主动发起补助申请，以便帮助未能自主申请但确有困难的学生。`

这条 Story 建议字段：

- Issue Type：Story
- Summary：`辅导员主动发起学生补助申请`
- Epic Link/Parent：`EP03 困难补助双入口申请`
- Assignee：成员 C
- Priority：Highest 或 High
- Story Points：5
- Due Date：2026-07-18
- Fix Version：`V1.0-课程验收版`
- Component：`补助管理`
- Labels：`backend` 可不填在 Story，更推荐填 `subsidy`、`tutor-entry`。

Description 可直接按下面填：

```text
用户故事：
作为辅导员，我希望为所管学生主动发起补助申请，
以便帮助未能自主申请但确有困难的学生。

范围：
1. 辅导员只能搜索自己所管学生。
2. 填写补助类型、金额、理由和附件。
3. 申请单记录实际发起人和发起方类型。
4. 提交后直接进入学院审批。
5. 学生收到通知并可查看进度。

验收条件：
- Given 辅导员已登录，When 搜索学生，Then 只显示所管学生。
- Given 表单填写完整，When 提交，Then 申请状态为待学院审批。
- Given 同批次已有有效申请，When 重复提交，Then 系统阻止并提示。
- Given 辅导员已提交，Then 学生能查看申请和进度。
```

### 第六步：拆分 Sub-task

上述 Story 可拆成：

1. `DB：补助申请增加发起方字段及索引`
2. `BE：所管学生查询和辅导员发起接口`
3. `BE：重复申请、权限和状态流转校验`
4. `FE：辅导员主动申请页面`
5. `FE：学生端申请详情与通知展示`
6. `TEST：双入口申请及越权测试`

如果一个人自己全栈完成，Sub-task 仍然有价值，因为能显示进度，也能防止漏掉测试和数据库修改。

## 6. Jira 字段到底怎么填

| 字段 | 填写方法 | 示例 |
| :--- | :--- | :--- |
| Issue Type | 按工作类型选 Epic/Story/Task/Sub-task/Bug | Story |
| Summary | 用动词+对象+结果，不要写“完成功能” | 学生提交爱心大礼包申请 |
| Description | 写背景、范围、验收条件、异常情况 | 参考上面的模板 |
| Assignee | 只填一个直接负责人 | 成员 B |
| Reporter | 创建或发现任务的人 | 建卡人 |
| Priority | Highest/High 用于阻塞主流程，Medium 用于普通功能，Low 用于美化 | 审批主流程=High |
| Story Points | 用1/2/3/5/8，表示相对复杂度 | 三级审批=8 |
| Due Date | 填真正需要完成的日期，不要全部都填24日 | 2026-07-19 |
| Epic Link/Parent | 挂到正确 Epic，子任务挂到 Story/Task | EP02 |
| Fix Version | 表示这个任务计划进入哪个版本 | V1.0-课程验收版 |
| Component | 功能归属，用于统计 | 绿色通道/补助/勤工助学 |
| Labels | 用于灵活筛选，不要每人自造一套词 | frontend、backend、db、test、core |
| Attachment | 上传原型、报错截图、接口示例或测试证据 | 截图/JSON/PDF |
| Linked Issues | 标注阻塞、依赖或相关任务 | C 的补助审请 blocks D 的看板统计 |

## 8. Scrum 和看板怎么用

### Sprint 设置

只有 9 天，推荐一个 Sprint：

- Sprint 名称：`Sprint 1 - 课程验收版`
- 开始：7 月 15 日
- 结束：7 月 23 日
- Sprint Goal：`完成绿色通道、补助双入口、辅导员事务、勤工助学和配套管理的可演示完整系统。`

7 月 24 日不再作为开发 Sprint 时间，作为验收缓冲日。

### 看板列

建议工作流：

```text
待办 To Do
  → 进行中 In Progress
  → 待代码审查 Code Review
  → 待测试 Testing
  → 待验收 Ready for Acceptance
  → 已完成 Done
```

如 Jira 不允许自定义这么多列，至少保留：`待办 → 进行中 → 待测试 → 完成`。

### 每列进入规则

- 进行中：已有明确负责人，且当天真正在做。
- Code Review：代码已提交并附有合并请求/提交记录。
- Testing：已部署到联调环境，测试人可操作。
- Ready for Acceptance：测试通过，已有截图或测试证据。
- Done：验收条件全部满足，代码已合并，文档已同步。

## 9. 四人每天怎么用 Jira

### 早上：15 分钟站会

每人只说三件事：

1. 昨天完成了哪些 Jira 卡。
2. 今天要把哪些 Jira 卡移到完成。
3. 现在被什么阻塞，需要谁帮助。

站会不现场讨论 30 分钟技术细节，有争议的人会后单独解决。

### 工作中

- 开始做前把卡移到 In Progress。
- 在评论中记录重要决定、接口变更和阻塞原因。
- 提交代码时在提交信息中写 Jira 编号，例如：`GCS-42 实现辅导员主动申请接口`。
- 任务发生范围变化时先更新 Jira，再继续开发。

### 晚上：10 分钟收尾

- 更新任务状态和剩余工作。
- 上传必要的截图或接口测试证据。
- 确认是否有卡片超过一天没有变化；如有，必须说明原因。

## 10. Bug 应该怎么填

Bug 标题不要只写“报错了”，应该写：

`[补助审批] 学院审批通过后可用额度未扣减`

Description 模板：

```text
环境：
- 版本/Commit：
- 浏览器：
- 账号角色：学院管理员

前置条件：
- 学院可用补助额度10000元
- 存在一条500元的待审申请

复现步骤：
1. 登录学院管理员账号
2. 进入补助审批详情
3. 填写建议金额500元并点击通过

实际结果：
- 审批成功，但可用额度仍为10000元

期望结果：
- 可用额度应变为9500元，且重复提交不应重复扣减

附件：
- 页面截图
- 请求与响应JSON
- 相关日志
```

Bug 优先级：

- Highest：系统无法启动、无法登录、数据丢失、严重越权、主流程完全中断。
- High：核心金额/状态错误、审批不能继续、重复发放。
- Medium：某个非核心操作失败，但有替代方式。
- Low：文案、对齐、颜色、非阻塞性交互问题。

## 11. Definition of Done（什么才算真正完成）

一条 Story 只有同时满足以下条件才能移到 Done：

- 验收条件全部通过。
- 前端、后端、数据库变更已合并。
- 正常流程、异常流程和权限校验至少各测一次。
- 关键操作有操作日志，状态变更有留痕。
- 没有 Highest/High 级未解决 Bug。
- Jira 中有截图、接口结果或测试记录作为证据。
- 与该功能相关的需求或数据库文档已同步。

## 13. 项目风险和当天处理规则

| 风险 | 预警信号 | 立即处理 |
| :--- | :--- | :--- |
| 公共框架阻塞所有人 | B/C/D 无法登录或无法使用公共接口 | A 停止次要页面，先修复公共阻塞；其他人用 mock 数据继续页面 |
| 审批状态不一致 | B/C 各自创建不同状态编码 | 16 日前锁定统一状态字典，修改需四人知情 |
| 数据库脚本冲突 | 多人同时修改同一 SQL | 每人提交独立变更脚本，A 每天合并一次 |
| 联调拖到最后 | 20 日仍只能单模块自测 | 每天晚上至少走一次跨模块流程 |
| Jira 沦为补作业 | 代码做完后才批量建卡 | 没有 Jira 卡不开始，开始时就移到进行中 |
| 一个人任务明显落后 | 连续两次日检查未达里程碑 | 将其中一条独立 Story 转给当前进度最快的成员 |

## 14. 向老师展示 Jira 时应该展示什么

1. Roadmap/路线图：展示 8 个 Epic 及 7 月 24 日的截止日期。
2. Backlog/待办：展示 Story 如何归属到 Epic、如何设置故事点和负责人。
3. Active Sprint/活动迭代：展示四人当前的进行中、待测试和已完成工作。
4. 一条写得完整的 Story：包含用户故事、验收条件、子任务和截图证据。
5. 一条 Bug：包含复现步骤、实际/期望结果、优先级和修复记录。
6. Version/Release：展示 `V1.0-课程验收版` 的完成比例。

这样老师能看到的不只是“我们创建了一些卡”，而是团队确实在用 Jira 做范围管理、分工、进度跟踪、测试和验收。
