## 3. 多模块地基

### 发现的问题

- 原根 `pom.xml` 会把后端打成一个可执行 JAR，B/C/D 无法把自己的模块作为统一父工程下的独立服务。
- D 的旧父 POM 使用 Spring Boot 3.2.x，与团队确认的 4.1.0 不一致。
- 各成员分支重复定义返回值、异常、鉴权或配置，直接合并会产生多套公共协议。

### 修正

- 根 POM 改为 `packaging=pom`，统一 Java 17、Spring Boot 4.1.0 和依赖版本。
- 建立 `common-api`、`platform-service`、`gift-service`、`subsidy-service`、
  `workstudy-service`、`dashboard-service`、`gateway-service`。
- 业务服务只依赖 `common-api`，不能通过 Maven 依赖其他业务服务的实现。
- 每个服务使用独立端口和环境变量数据库配置，根目录不再保留可执行 `src`。

### 验证

- PR #2 合入时 20 个测试通过；所有模块可由根 Reactor 构建。

## 4. B：绿色通道与礼包模块

### 原分支可复用内容

批次、礼包物品、配额、学生申请的实体、Mapper、Service 和 CRUD 控制器可以迁入 `gift-service`。

### 发现的问题

1. 原 `/api/gift/apply/pickup` 直接接收客户端传来的整条 `StudentApply`，然后执行 `updateById`。
   客户端可以修改不应修改的字段，接口也没有按领取码核验。
2. 原核销没有检查“待领取”状态，重复请求或两名工作人员并发操作时可能重复领取。
3. MyBatis 配置类位于业务启动类默认扫描范围之外，文件搬入子模块后可能没有分页插件、自动填充等配置。
4. 原分支没有覆盖领取码不存在、已经领取、并发重复核销等测试。

### 修正

- 新增只包含 `pickupCode`、`operatorId`、`remark` 的 `PickupRequest`，不再允许客户端覆盖整个申请。
- 服务端按领取码查询；领取码为空返回 `40000`，不存在返回 `40400`，状态冲突返回 `40900`。
- 更新 SQL 同时限定 `id` 和 `pickup_status=待领取`，利用条件更新防止并发重复核销。
- 把 MyBatis 配置纳入 `edu.greenchannel` 扫描范围，并统一到父 POM 管理的依赖版本。
- 增加 3 个核销服务测试，覆盖成功、重复领取和并发状态变化。

### 结果

- PR #3：23 个累计测试通过，`gift-service` 可独立打包。

## 5. C：困难补助模块

### 原分支可复用内容

补助批次、学院/年级额度、学生与辅导员双入口申请、分级审批和查询逻辑可以迁入
`subsidy-service`。

### 发现的问题

1. 额度接口相信客户端的 `X-User-Role` 请求头，并提供默认角色；调用者可以伪造角色提升权限。
2. 各服务若各自保存内存 Token，platform 登录产生的令牌无法在 subsidy 等服务中解析。
3. 查询和审批需要按当前用户、学院范围过滤，不能只依赖前端传入的 ID。
4. C 的代码仍位于旧根单体目录，并夹带对 A 用户模型和数据库脚本的修改，不能整棵目录覆盖。

### 修正

- 将 `AuthInterceptor`、`PermissionInterceptor`、`CurrentUser`、`RequirePermission` 和 `TokenService`
  统一放入 `common-api`。
- Token 改为带 HMAC 签名、8 小时过期的无状态令牌；所有服务共享
  `APP_AUTH_TOKEN_SECRET` 后即可跨服务验签。
- 控制器从服务端解析出的 `CurrentUser` 取用户和角色，不再信任 `X-User-Role` 等身份请求头。
- 增加权限注解、学院范围和当前学生范围校验，并为额度业务补测试。

### 结果

- PR #4：27 个累计测试通过。

## 6. D：勤工助学与第一版看板

### 原分支可复用内容

岗位、批次、报名、面试、录用、考勤、薪酬、评价、协议以及看板/报表的主要业务类可以迁移。

### 发现的问题

1. 勤工控制器把 `studentId`、`userId`、`approverId`、`interviewerId` 作为请求参数，调用者可以冒充他人。
2. 多个服务直接抛 `RuntimeException`，会被统一异常处理当作 500，前端无法区分参数、权限和状态冲突。
3. D 使用自定义 `Result`，与 `common-api.ApiResponse` 重复。
4. 协议生成时没有可靠写入学生和岗位关联，后续签署可能无法校验归属。
5. D 的配置文件包含个人 MySQL 明文密码；旧 POM 和包名也与 Boot 4.1.0 聚合工程不一致。
6. 自定义报表允许动态字段，若不做字段白名单会形成 SQL 注入入口。

### 修正

- 所有身份相关控制器从 `CurrentUser` 读取当前用户；审批、面试等接口增加权限要求。
- 将可预期业务失败转换为带 `40000/40300/40400/40900` 的 `BusinessException`。
- 全部响应统一为 `ApiResponse`，包名统一为 `edu.greenchannel.*`。
- 修复协议的 `studentId`、`positionId` 关联和签署校验。
- 未迁入明文密码；数据库凭据只使用 `DB_PASSWORD` 等环境变量。
- 为报表字段建立白名单；增加身份绑定、协议生成和字段注入测试。

### 结果

- PR #5：32 个累计测试通过；`workstudy-service` 与 `dashboard-service` 均可打包。

## 7. D 后续看板提交的二次整合

D 在第一轮迁移期间又向 `dev/kang` 追加了模块注册、统一报表和 WebSocket 推送设计。设计方向可用，
但最新代码不能原样合入。

### 发现的问题

1. `DynamicQueryMapper` 使用 `@Select("${sql}")` 执行整段动态 SQL，筛选值又通过字符串拼接加入，
   `batchId/year/month/collegeId/status` 都可能成为 SQL 注入入口。
2. 公共 WHERE 构造器在不同报表中混用 `p`、`s`、`sal` 别名；薪酬查询还会形成重复 `WHERE`。
3. 岗位明细把考勤表当成存在 `position_id`，实际实体和表使用 `hire_id`，查询会直接报错。
4. 后续配置再次出现个人 MySQL 明文密码。
5. 手动推送接口没有权限约束，导出文件名也未清理换行和非法字符。

### 修正

- 保留 `ModuleDataService`、`ModuleRegistry` 和按模块推送的设计，不保留整段动态 SQL 执行器。
- 为岗位、学生、薪酬、岗位明细分别写固定 SQL；筛选值先解析为数字，再使用 `#{...}` 参数绑定。
- 报表字段继续经过模块/报表类型双重白名单，返回结果在 Java 中投影到获准字段。
- 考勤和薪酬明细统一按 `hire_id` 关联；修复表别名和 WHERE 组合。
- 手动推送要求 `school:dashboard:view`；导出文件名清理 CR/LF、路径和系统非法字符。
- 明文密码仍未迁入，并补充恶意 `batchId="1 OR 1=1"` 被拒绝的测试。

### 结果

- PR #7：37 个累计测试通过，MyBatis XML 可解析，8 个模块全部打包成功。

## 9. TECH-T06：前后端唯一仓库与统一入口

### 来源

- 前端取自原仓库 `YuCPbit/green-channel-frontend` 的 `main@04224e5`，包含 A 的登录菜单框架和 C 的补助页面。
- 导入后统一放在本仓库 `frontend/`；原前端仓库不再作为后续集成目标。

### 发现的问题

1. 前后端仍分属两个 GitHub 仓库，不符合老师确认的“只有一个仓库”。
2. platform-service 单独维护 CORS，其他服务没有同一策略；前端若只走网关，不应由业务服务分别处理跨域。
3. 补助前端仍发送 `X-User-Role`、`X-College-Id` 自报身份头，与统一服务端鉴权冲突。
4. 前端锁定的 Vite 7.0.4 命中已公开的高危开发服务器漏洞。

### 修正与验证

- 将前端完整导入 `frontend/`，保留 Vite `/api` → 8080 的开发代理，并更新唯一仓库启动说明。
- CORS 收口到 `gateway-service`；默认允许 `http://localhost:5173`，其他来源通过
  `APP_CORS_ALLOWED_ORIGINS` 配置。platform-service 删除重复 CORS。
- 补助请求只携带 Bearer Token，不再发送客户端自报角色或学院。
- Vite 升级并锁定到 7.3.6；`npm audit` 为 0 个漏洞，`npm run build` 成功。
- 网关增加真实预检响应头测试；全后端累计 38 个测试通过。

## 10. 明确没有原样合入的内容

| 未合入内容 | 原因 | 替代方案 |
| --- | --- | --- |
| 成员分支自己的根/父 POM | 版本和聚合结构冲突 | 统一使用根 Boot 4.1.0 父 POM |
| 重复的 `Result`、异常和鉴权类 | 会产生多套协议 | 使用 `common-api` |
| 客户端身份请求参数/角色请求头 | 可冒充或越权 | `CurrentUser` + 权限注解 |
| 明文数据库密码 | 凭据泄露 | 环境变量；已出现过的密码应轮换 |
| 整段 `${sql}` 动态查询 | SQL 注入且难测试 | 固定 Mapper SQL + 参数绑定 + 字段白名单 |

## 11. 当前仍然存在的风险和待办

1. 37 个自动测试和打包已经通过，但尚未在本机同时连接真实 MySQL 启动六个服务并跑完整业务冒烟。
2. 无状态 Token 的 `logout` 当前只要求客户端丢弃令牌；服务端不能立即吊销，最迟 8 小时自然失效。
3. dashboard 当前会直接读取多个业务域的共享数据库表，后续若部署为真正独立数据库，需要改成服务接口或事件汇总。
4. GitHub 当前没有强制 CI 检查，PR 页面显示 0 个自动 Checks；应补 Maven CI 和密钥扫描。
5. B 的绿色通道前端、D 的勤工/看板前端、部署脚本和五角色端到端验收仍未完成。
6. D 的历史分支曾提交明文数据库密码。即使 `main` 没有该密码，也应视为已经暴露并执行轮换。
