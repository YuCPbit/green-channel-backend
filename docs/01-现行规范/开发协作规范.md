# 开发协作规范

> 状态：现行，团队必须遵循
> 最后核验：2026-07-22
> 维护人：A（公共架构）；涉及模块变更时由对应成员共同更新

## API 约定

- 基础路径统一为 `/api`，JSON 使用 UTF-8。
- 成功响应：`{"code":0,"message":"success","data":{},"timestamp":"..."}`。
- 业务错误使用明确错误码；未登录统一为 `40100`，系统异常统一为 `50000`。
- 登录后的请求携带 `Authorization: Bearer <token>`。
- 当前使用 HMAC 签名的无状态令牌。所有服务必须配置同一个至少 32 字符的
  `APP_AUTH_TOKEN_SECRET`；控制器不得相信客户端自行传入的用户 ID 或角色请求头。
- 前端只能通过网关 `/api` 调用后端，不得直连 8081—8085，也不得发送 `X-User-Role`、
  `X-College-Id` 等自报身份头。跨域策略统一由 `gateway-service` 管理。

## Git 约定

- 主分支：`main`，只合并已自测的代码。
- 本仓库 `main` 是后端唯一集成基线；前端以 `green-channel-frontend/main` 为基线。`dev/kang` 等历史个人分支只用于追溯，不得整体合回后端 `main`。
- 新工作必须从最新 `main` 创建分支；需要复用历史分支时只迁移经审查的增量代码。
- 功能分支：`feature/GCS-编号-简短说明`。
- 缺陷分支：`fix/GCS-编号-简短说明`。
- 提交格式：`类型(GCS-编号): 简短说明`，例如 `feat(GCS-S01): 完成五类用户登录`。
- 类型使用 `feat`、`fix`、`docs`、`test`、`refactor`、`chore`。
- 合并前至少完成构建、正常流程、异常流程和权限检查。

## 数据库约定

- 表名以 `gc_` 开头，使用逻辑删除字段 `is_deleted`。
- 新增数据库变更必须提供可重复执行的迁移 SQL，并同步《数据库设计说明》。
- 不在业务代码中硬编码字典项中文名称；稳定的系统角色编码除外。
- 禁止把个人数据库密码、身份证号明文或真实学生数据提交到仓库。

## 首日接口

- `GET /api/health`：检查应用和数据库连接。
- `POST /api/auth/login`：账号密码登录。
- `GET /api/auth/me`：获取当前用户及其菜单。
- `POST /api/auth/logout`：注销当前令牌。
